# Codex Context

## Zsh history policy

- Keep `INC_APPEND_HISTORY` enabled so commands are saved to the shared
  `~/.zsh_history` file promptly, even from long-lived tmux panes.
- Keep `SHARE_HISTORY` disabled so existing tmux panes, windows, and sessions do
  not import each other's live commands into `up-line-or-history`, `Ctrl-P`, or
  fzf-backed `Ctrl-R` navigation.
- If a live shell needs to import newly saved global history explicitly, use
  zsh history import commands such as `fc -RI` rather than enabling automatic
  live sharing.

## Zsh startup organization

- Keep `zsh/.zshrc` focused on interactive shell orchestration: guard
  non-interactive shells, prepare zsh cache paths, set key bindings, and source
  modular config files.
- Keep exported environment variables and path segments in `zsh/.exports`.
  This includes Homebrew environment setup so `HOMEBREW_PREFIX` exists before
  completions initialize.
- Keep shell behavior options in `zsh/.options`; do not scatter `setopt` or
  `unsetopt` calls through environment files.
- In `zsh/.completions`, update `fpath` before running `compinit`. Paths added
  after `compinit` are too late for the initial completion dump.
- Avoid top-level `local` or bare `typeset name` declarations in sourced zsh
  modules. When the variable is already set, zsh prints `name=value` during
  startup; use direct assignment at top level and reserve `local` for functions.

## Vercel MCP via Codex

- Codex direct remote MCP OAuth for `https://mcp.vercel.com` can fail before
  auth with Vercel's "app redirect URL is invalid" page because Codex uses a
  localhost callback shape Vercel rejects.
- `mcp-remote` avoids that redirect shape, but its bundle changes Node 26 global
  fetch behavior; Vercel OAuth metadata then arrives as compressed bytes and
  discovery fails before auth.
- Use `codex/bin/vercel-mcp-remote.mjs` through `npx --package mcp-remote` for
  the Codex `vercel` MCP server. The wrapper swaps global fetch to
  `mcp-remote`'s bundled `undici.fetch` before importing the real proxy.
- Keep `startup_timeout_sec` high enough for the first browser OAuth approval;
  120 seconds is the current local setting.

## Clerk MCP via Codex

- Clerk's MCP server is a streamable HTTP endpoint. Configure it with
  `codex mcp add clerk --url https://mcp.clerk.com/mcp` or a
  `[mcp_servers.clerk] url = "https://mcp.clerk.com/mcp"` stanza.
- If Codex reports `MCP client for clerk failed to start: No such file or
  directory`, check for an incorrect `command = "https://mcp.clerk.com/mcp"`
  entry. That makes Codex treat the URL as a local executable.

## Codex Skill Budget

- Codex loads cross-agent skills from `~/.agents/skills`; keep the GSD skill
  source there instead of duplicating the same `gsd-*` directories under
  `~/.codex/skills`.
- Keep `superpowers@openai-curated` and `expo@openai-curated` enabled globally.
  They are expected in normal Codex sessions for workflow discipline and iOS
  development.
- Avoid leaving legacy duplicate Superpowers skills under
  `~/.codex/superpowers/skills` when the OpenAI-curated Superpowers plugin is
  enabled.
- If Codex shows the skill-description budget warning, first check for duplicate
  skill roots and stale per-skill disables whose plugin version changed.

## OpenCode MCP environment

- OpenCode resolves remote MCP URLs before attempting OAuth or bearer-token
  auth. If `opencode mcp list` reports `Invalid MCP URL for "supabase"`, check
  that the launching shell has `SUPABASE_MCP_URL`.
- The OpenCode `supabase` MCP config sends `Authorization: Bearer
  {env:SUPABASE_ACCESS_TOKEN}`. If the URL is valid but OpenCode reports
  `needs authentication`, check the local `SUPABASE_ACCESS_TOKEN` value.
- On this machine, private OpenCode MCP env values belong in `~/.exports_local`;
  do not commit concrete project URLs or access tokens into this public repo.

## Public repo hygiene

- Keep local editor/app state out of the public index. Exported Cursor profiles
  should stay ignored via `cursor/*.code-profile`.
